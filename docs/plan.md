# SecAgent — AI安全智能分析Agent平台

## 综合课程设计项目方案

---

## 一、项目背景与目标

### 1.1 项目背景

随着网络攻击手段日益复杂化，传统安全工具依赖规则匹配和人工分析的模式面临巨大挑战：漏洞数量爆发式增长、恶意代码变种层出不穷、安全分析师严重不足。大语言模型（LLM）的崛起为安全领域带来了新的可能性——AI不仅能理解代码语义、识别隐蔽的攻击模式，还能像人类分析师一样进行多步骤推理和决策。

本项目旨在构建一个**具备工具调用能力的AI安全分析Agent**——SecAgent，将LLM的语义理解能力与传统安全工具的高效检测能力相结合，面向两大核心场景：

- **智能漏洞检测**：自动分析源代码，发现潜在安全漏洞并给出修复建议
- **恶意代码分析**：对可疑文件进行静态/动态分析，判定恶意性并提取威胁指标

### 1.2 项目目标

| 目标维度           | 具体描述                                                            |
| ------------------ | ------------------------------------------------------------------- |
| **功能目标** | 实现代码漏洞自动检测与恶意文件智能分析，输出结构化安全报告          |
| **技术目标** | 构建基于ReAct范式的Agent引擎，集成静态分析工具链与LLM推理能力       |
| **展示目标** | 提供精美的Web交互界面，实时可视化Agent的思考与分析过程              |
| **学习目标** | 深入理解LLM Function Calling、Agent架构设计、安全工具集成等核心技术 |

### 1.3 预期价值

- **教育价值**：可作为安全课程教学工具，帮助学生学习漏洞原理
- **实用价值**：辅助开发者进行代码安全自查，提升软件安全性
- **研究价值**：探索LLM在安全分析中的能力边界与最佳实践

### 2.2 核心交互流程

```
用户提交代码/文件
   │
   ▼
Agent接收任务 → 识别任务类型（漏洞检测 or 恶意分析）
   │
   ▼
进入ReAct循环（最多10轮）：
   │
   ├─ Thought: LLM分析当前状态，推理下一步该做什么
   ├─ Action: LLM决策调用哪个工具 + 参数
   ├─ Observe: 执行工具，获取结果反馈给LLM
   │           ↓ 每步通过WebSocket推送给前端
   │           前端渲染思维链卡片（思考→动作→观察）
   │
   ▼
分析完成 → 聚合所有结果 → 生成结构化报告
   │
   ▼
前端展示报告（漏洞列表/恶意判定 + 图表 + 建议）
```

## 三、AI技术选型

### 3.1 核心AI技术说明

> **面向本科生**：以下AI技术均为"调用现有API服务"的方式使用，不需要从零训练模型或涉及复杂数学推导。核心工作在于：设计Promt（提示词）、定义工具接口（Function Schema）、编排Agent逻辑流程。

### 3.2 LLM模型选择

| 用途            | 推荐模型          | 原因                                               |
| --------------- | ----------------- | -------------------------------------------------- |
| Agent主推理引擎 | 通义千问 Qwen-turbo | 支持Function Calling、推理能力强、中文友好、价格低 |
| 备选方案        | 通义千问 Qwen-Max   | 阿里云生态、国内访问稳定、安全合规                 |

### 3.3 Agent推理范式：ReAct循环

ReAct（Reasoning + Acting）是目前最主流的Agent设计模式，核心思想是让LLM**交替进行推理和行动**：

```
┌──────────────────────────────────────────────┐
│              ReAct 循环示意图                  │
│                                               │
│  ┌──────────┐     ┌──────────┐               │
│  │ THOUGHT  │────▶│  ACTION  │               │
│  │ 我应该先  │     │ 调用semgrep│              │
│  │ 进行静态  │     │ 扫描代码   │              │
│  │ 扫描     │     │           │              │
│  └──────────┘     └─────┬─────┘               │
│         ▲               │                     │
│         │        ┌──────▼──────┐              │
│         │        │  OBSERVE    │              │
│         └────────│  扫描发现   │              │
│   根据结果思考    │  3个可疑点  │              │
│   下一步该做什么  └─────────────┘              │
│                                               │
│   循环直到：任务完成 或 达到最大步数(10步)       │
└──────────────────────────────────────────────┘
```

### 3.4 Function Calling（函数调用）

这是LLM能"使用工具"的关键技术。核心做法：

1. **定义工具Schema**：用JSON描述每个工具的用途和参数
2. **LLM决定调用**：AI判断当前需要哪个工具，生成调用参数
3. **服务端执行**：后端实际运行工具，将结果返回给AI
4. **AI继续推理**：基于工具返回结果，决定下一步

示例——定义一个"代码扫描"工具：

```json
{
  "name": "scan_code",
  "description": "使用semgrep对代码进行静态安全扫描",
  "parameters": {
    "code": {"type": "string", "description": "待扫描的源代码"},
    "language": {"type": "string", "enum": ["python", "java", "javascript"]}
  }
}
```

### 3.5 Prompt工程

我们需要为Agent设计专业的System Prompt（系统提示词），核心要素：

- **角色定位**：你是一名资深安全分析专家，擅长代码审计和恶意代码分析
- **能力边界**：明确告知Agent可以使用哪些工具及各自用途
- **输出规范**：要求按照结构化格式输出，便于解析
- **安全约束**：不执行危险操作、不确定时标注置信度

### 4.1 功能总览

```

│   ├── 静态规则扫描（semgrep + bandit）

├── 功能二：恶意代码/文件分析 ★核心
│   ├── 文件上传（PE/ELF/脚本/Office文档）
│   ├── 文件特征提取与静态分析
│   ├── AI行为逻辑分析（代码语义理解）
│   ├── IOC提取（IP/域名/URL/Hash）
│   ├── 威胁情报关联查询

│   ├── 分析过程中用户可追问

    ├── 结构化Markdown报告

    ├── PDF导出
    └── 历史任务管理
```

### 4.2 功能一：智能代码漏洞检测（详细）

**分析流程：**

```
源代码输入
  → 语言识别
  → 静态规则扫描（semgrep/bandit）
  → 获取初步告警列表（含大量误报）
  → AI逐条研判：
      ├── 真漏洞：确认类型/CWE编号/危险等级
      ├── 误报：排除并说明原因
      └── 不确定：标注置信度
  → 查询CVE漏洞库获取相关案例
  → 生成修复建议（含代码diff示例）
  → 输出结构化漏洞报告
```

**检测的漏洞类型：**

- SQL注入（CWE-89）
- 跨站脚本XSS（CWE-79）
- 命令注入（CWE-78）
- 路径遍历（CWE-22）
- 硬编码密钥（CWE-798）
- 反序列化漏洞（CWE-502）
- 权限控制缺陷（CWE-862）

**Agent的自主决策点：**

- 选择先扫描还是先进行代码结构分析
- 判断哪些告警需要深入分析
- 决定是否需要查询CVE库获取上下文

### 4.3 功能二：恶意代码/文件分析（详细）

**分析流程：**

```
文件上传
  → 文件类型识别（PE/ELF/脚本/Office/其他）
  → 文件基本信息提取（大小/Hash/类型/时间戳）
  → 根据文件类型选择分析路径：
      ├── PE文件：导入表分析、字符串提取、节区信息
      ├── 脚本文件：代码格式化、混淆检测、AI语义分析
      ├── Office文档：宏提取、OLE对象分析
      └── 其他：通用特征提取
  → 静态特征提取（可疑字符串/IP/URL/Base64编码内容）
  → AI行为分析（理解代码意图、判断恶意行为模式）
  → IOC（失陷指标）提取
  → 威胁情报查询（IP/域名/Hash是否已知恶意）
  → ATT&CK攻击技术映射
  → 恶意判定：恶意 / 可疑 / 安全（含置信度）
  → 输出分析报告
```

**Agent的自主决策点：**

- 根据文件类型选择分析策略
- 判断是否需要解码/反混淆处理
- 决定提取哪些IOC去查询威胁情报

### 4.4 功能三：交互式安全分析对话

支持用户在分析过程中或分析完成后追问：

- "帮我看一下第42行的SQL查询有没有问题"
- "这个文件的网络行为具体做了什么"
- "给出的修复方案有没有替代做法"

Agent基于分析上下文和历史对话记忆进行回复。

### 4.5 功能四：报告生成

**报告结构：**

```
# 安全分析报告
## 基本信息（分析时间、目标、耗时）
## 分析摘要（Agent思考链摘要）
## 详细发现（漏洞/恶意行为列表，含等级/类型/代码片段）
## 可视化图表（风险分布饼图、ATT&CK热力图）
## 修复建议（按优先级排序，含代码示例）
## IOC清单（仅恶意代码分析）
## 参考链接（CVE/CWE/ATT&CK详情页）
```

## 五、数据来源与处理

### 5.1 数据来源

| 数据类型     | 具体来源                          | 使用方式         | 用途             |
| ------------ | --------------------------------- | ---------------- | ---------------- |
| 用户提交数据 | Web界面上传/粘贴                  | 直接分析         | 分析对象         |
| 漏洞数据库   | [NVD CVE API](https://nvd.nist.gov/) | API查询          | 漏洞信息关联     |
| 弱点分类库   | MITRE CWE列表                     | 本地JSON文件缓存 | 弱点分类参考     |
| ATT&CK知识库 | MITRE ATT&CK STIX数据             | 本地JSON文件缓存 | 攻击技术映射     |
| 威胁情报     | AlienVault OTX / URLhaus          | API查询          | IOC信誉查询      |
| 静态分析规则 | semgrep Registry / bandit内置     | 本地安装         | 代码规则扫描     |
| YARA规则     | 开源社区规则集                    | 本地加载         | 恶意文件特征匹配 |

### 5.2 数据处理流程

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  输入层       │    │  分析层       │    │  输出层       │
│              │    │              │    │              │
│ 代码文件  ───┼───▶│ 格式识别     │───▶│ 漏洞报告     │
│ 二进制文件───┼───▶│ 特征提取     │───▶│ 恶意分析报告 │
│ 文本粘贴  ───┼───▶│ Agent分析   │───▶│ 可视化图表  │
│              │    │ 报告生成     │    │ PDF文档     │
└──────────────┘    └──────────────┘    └──────────────┘
```

### 5.3 数据存储设计

由于是原型系统，采用轻量级方案：

- **分析任务**：SQLite task表（id, type, status, input_path, result_json, created_at）
- **分析结果**：JSON格式存储完整分析结果，便于前端渲染
- **静态参考数据**：CWE列表、ATT&CK矩阵等以JSON文件形式存放在项目中

## 七、技术难点与解决方案

### 7.1 技术难点一览

| 序号 | 难点                   | 风险等级 | 影响                |
| ---- | ---------------------- | -------- | ------------------- |
| 1    | AI分析准确性不足       | 🔴 高    | 漏报/误报影响可信度 |
| 2    | Function Calling不稳定 | 🟡 中    | Agent调用错误工具   |
| 3    | 长分析任务耗时         | 🟡 中    | 用户体验差          |
| 4    | API调用成本控制        | 🟡 中    | 开发测试费用超预算  |
| 5    | 恶意文件安全风险       | 🟡 中    | 分析系统被感染      |
| 6    | 前端实时展示技术复杂度 | 🟢 低    | 实现难度            |
| 7    | Prompt工程调优         | 🟡 中    | 影响分析效果        |

### 7.2 详细解决方案

#### 难点1：AI分析准确性不足

**问题描述**：LLM可能出现"幻觉"，对不存在的漏洞信誓旦旦，或遗漏真正的安全问题。

**解决方案：**

1. **双重验证机制**：静态工具扫描 + AI研判，两者结果交叉验证。静态工具的发现AI不能直接推翻（只能降级为"需人工确认"），AI发现的新问题必须在代码中找到确切证据
2. **置信度标注**：每次分析结果强制标注置信度（高/中/低），置信度低的结论在报告中显著标明
3. **证据链要求**：System Prompt要求AI输出结论时必须引用具体代码行号和上下文
4. **角色限定**：Prompt明确约束"你的职责是辅助分析而非最终决策"，避免过度自信

#### 难点2：Function Calling不稳定

**问题描述**：LLM可能调用错误的工具、传入格式错误的参数、或跳过必要的工具调用步骤。

**解决方案：**

1. **严格的参数校验**：服务端对所有工具调用参数进行JSON Schema校验，不合法参数立即拒绝并反馈错误信息
2. **重试机制**：每个工具调用最多允许3次重试，每次重试时将错误信息反馈给LLM
3. **兜底策略**：如果Agent连续3步没有实质进展，自动切换为预设的标准分析流程
4. **工具描述优化**：Schema中提供清晰的参数示例和使用场景说明

#### 难点3：长分析任务耗时

**问题描述**：复杂样本的Agent分析可能需要多轮推理+工具调用，总耗时可能较长。

**解决方案：**

1. **异步处理**：任务提交后立即返回task_id，后台异步执行
2. **实时进度推送**：WebSocket推送每一步进展，让用户感知进度
3. **超时与步数限制**：单任务最多10个ReAct步骤，总超时5分钟
4. **过程即展示**：将"等待"转化为"观看Agent思考"的体验，成为产品亮点

#### 难点4：API调用成本控制

**问题描述**：每次Agent分析可能调用LLM多次，Token消耗累积。

**解决方案：**

1. **分析结果缓存**：相同代码Hash的分析结果直接复用
2. **分级调用策略**：简单判断（如文件类型识别）使用规则而非LLM
3. **Token预算管理**：单任务设置Token上限（如50K tokens）
4. **开发阶段控制**：使用通义千问（价格低）进行开发测试，成本可控

#### 难点5：恶意文件安全风险

**问题描述**：上传的真实恶意样本可能危害分析系统。

**解决方案：**

1. **静态分析优先**：默认仅进行静态分析，不执行任何上传文件
2. **Docker隔离**：动态分析功能在独立Docker容器中执行，分析完成后销毁
3. **文件限制**：上传文件大小限制50MB，类型白名单控制
4. **临时目录处理**：所有文件在临时目录处理，分析完成后自动清理
5. **禁止网络访问**：沙箱容器禁止出站网络

#### 难点6：前端实时展示

**问题描述**：需要将Agent复杂的思考过程以直观、动态的方式展示。

**解决方案：**

1. **标准化数据格式**：定义统一的"思维步"数据结构
2. **WebSocket事件驱动**：每个分析步骤触发一个事件，前端响应渲染
3. **状态机驱动动画**：前端使用状态机管理展示状态（思考中→执行中→观察中→完成）
4. **渐入动画**：每个思维卡片依次渐入，模拟"AI在思考"的效果

#### 难点7：Prompt工程调优

**问题描述**：不好的Prompt会导致Agent行为不符合预期。

**解决方案：**

1. **模块化Prompt设计**：分为角色设定、工具说明、输出格式、约束规则等模块，便于分别调优
2. **A/B测试**：准备3-5个测试样本，对比不同Prompt版本的分析效果
3. **Few-shot示例**：在Prompt中提供1-2个完整的分析示例
4. **持续迭代**：开发过程中每个阶段根据实际效果调整Prompt

## 八、使用场景

### 场景1：课程代码安全审查

> **操作**：提交课程设计Python代码

>
> 1. 识别代码语言为Python
> 2. 调用bandit进行Python专项扫描
> 3. 发现一处SQL查询使用字符串拼接
> 4. AI深入分析上下文，确认存在SQL注入风险
> 5. 查询CWE-89获取漏洞详情
> 6. 生成修复建议：使用参数化查询替代字符串拼接
>    **输出**：包含漏洞位置（文件名+行号）、风险等级（高危）、CWE编号、PoC、修复代码示例的完整报告

### 场景2：可疑邮件附件分析

> **角色**：普通用户/安全运维
> **操作**：收到可疑邮件，上传 `.docm`附件
> **Agent执行**：
>
> 1. 识别为Office宏文档
> 2. 提取文档中的VBA宏代码
> 3. AI分析宏代码逻辑：发现从远程URL下载可执行文件并运行
> 4. 提取IOC：C2域名 `evil-server.example.com`
> 5. 查询威胁情报：该域名在URLhaus标记为恶意
> 6. 映射ATT&CK技术：T1204（用户执行）+ T1105（远程文件拷贝）
>    **输出**：判定为"恶意文件"（置信度98%），附带行为分析、IOC清单、ATT&CK映射

### 场景3：Web应用安全评估

> **角色**：开发者
> **操作**：提交一个Flask/Django项目的源代码包
> **Agent执行**：
>
> 1. 遍历项目文件树，识别所有Python源文件
> 2. 对每个文件依次进行semgrep扫描
> 3. 汇总告警，AI逐条研判去重
> 4. 发现：3个XSS漏洞、2个SQL注入、1个路径遍历
> 5. AI按危险等级排序：SQL注入 > 路径遍历 > XSS
> 6. 为每个漏洞生成修复方案
>    **输出**：安全评估总览（风险分布饼图 + 问题清单表格 + 优先级排序 + 修复方案）

### 场景4：交互式深度分析

> **角色**：安全研究员
> **操作**：上传代码后，在分析过程中追问
> **对话示例**：
>
> ```
> 用户：上传 app.py（一个Flask应用）
> Agent：[自动分析] 发现3个潜在安全风险...
> 用户："重点看一下登录逻辑有没有问题"
> Agent：定位到 login() 函数
>       [分析] 发现密码使用MD5（已不安全）+ 缺少登录失败速率限制
>       [建议] 使用bcrypt替换MD5 + 添加登录尝试次数限制
> 用户："怎么用bcrypt？给个代码示例"
> Agent：[生成] 完整的Flask+bcrypt密码哈希集成代码示例
> ```

### 场景5：恶意脚本行为剖析

> **角色**：安全运维人员
> **操作**：上传一个混淆的JavaScript文件
> **Agent执行**：
>
> 1. 检测到代码高度混淆（熵值 > 7.5）

> 3. 还原出：窃取Cookie + WebSocket隧道 + 键盘记录
> 4. 提取C2通信地址
> 5. 分析完整的攻击链
>    **输出**：去混淆后的行为描述 + 攻击链图 + IOC + 清除建议

### 场景6：项目汇报展示

> **角色**：评审老师
> **操作**：观看演示流程
> **演示设计**：
